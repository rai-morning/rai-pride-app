# 公開・運用チェックリスト（0円〜3,000円/月）

このドキュメントは、現在の `Next.js + Firebase` 構成を前提に、
- 公開前にやること
- セキュリティと運用の最低ライン
- ユーザー増加時のインフラ判断
- 月額 0円〜3,000円で収めるための数値ルール

を1つにまとめた実務用チェックリストです。

---

## 1) 0円運用チェック（公開前）

- [ ] Vercel は Hobby（無料）で開始する（まずは `*.vercel.app` を使う）
- [ ] Firebase は Spark（無料）で開始し、課金プランは有効化しない
- [ ] Vercel / Firebase 両方で使用量アラート通知を設定する
- [ ] 独自ドメイン購入は後回し（必要時のみ）
- [ ] 画像・動画アップロードに上限ルールを入れる（無制限禁止）
- [ ] 不要ログの長期保存をしない
- [ ] デプロイ対象ブランチを絞る（`main` 中心）
- [ ] `next dev` ではなく本番環境（デプロイ先）で最終確認する

---

## 2) セキュリティ・設定チェック

- [ ] Firestore / Storage ルールを最小権限で定義する
- [ ] 認証必須ページ/APIを明確化する
- [ ] ログインや投稿系にレート制限を入れる
- [ ] 環境変数に秘密情報を置き、クライアント側に不要な秘匿値を出さない
- [ ] 監査ログ確認手順（誰が何をしたか追える）を整備する

---

## 3) ユーザー増加を見据えたインフラ検討チェック

- [ ] 想定DAU/同時接続数を定義する（例: DAU 1,000 → 10,000）
- [ ] Firestore の read/write 負荷を計測する
- [ ] 重い一覧/検索クエリを特定し、インデックスを最適化する
- [ ] キャッシュ戦略（ISR, CDN）を決める
- [ ] 通知/集計など重い処理を非同期化する
- [ ] 監視・アラート・障害復旧手順を決める
- [ ] 月次コスト上限と超過時アクションを決める
- [ ] バックアップ/復旧のリハーサルを実施する

---

## 4) 「今のサーバーで大丈夫か？」判断基準

結論: 現時点の構成（Next.js + Firebase）で問題なし。  
ただし、次の症状が継続したら見直しタイミング。

- [ ] Firestore 読み取り費用が毎月増加し、主要コストになっている
- [ ] 一覧表示の体感レスポンスが 1〜2秒超で常態化している
- [ ] メディア配信コストがDBコストを上回っている
- [ ] サーバー処理待ちがUXを悪化させている
- [ ] 無料枠超過が毎月継続している

---

## 5) 月額 0円〜3,000円で収める運用ルール表（数値入り）

以下は「まずは小さく公開して安全に育てる」ための現実的な運用基準です。  
厳密な請求額は契約プラン・トラフィックで変わるため、毎月実績で補正してください。

### 5-1. 月額予算ルール

| 項目 | 目標 | 警戒ライン | 上限ライン（対応必須） |
|---|---:|---:|---:|
| 総額（Vercel + Firebase + その他） | 0〜1,000円 | 1,500円 | 3,000円 |
| Firebase 請求 | 0〜800円 | 1,200円 | 2,000円 |
| Vercel 請求 | 0円 | 500円 | 1,000円 |
| ドメイン費（任意） | 0円（未導入） | - | 年払い時のみ別管理 |

運用ルール:
- 警戒ライン到達時: 新機能追加より先にコスト要因を特定
- 上限ライン到達時: 画像圧縮・読み取り削減・キャッシュ導入を優先実施

### 5-2. トラフィック/利用量ルール（初期目安）

| 指標 | 目標 | 警戒ライン | 上限ライン |
|---|---:|---:|---:|
| DAU（1日アクティブ） | 〜300 | 500 | 1,000 |
| 1ユーザーあたり画面遷移/日 | 〜30 | 50 | 80 |
| 1画面表示あたりFirestore read | 1〜5 | 6〜10 | 11以上 |
| 1ユーザーあたりwrite/日 | 〜20 | 30 | 50 |
| 画像アップロード平均サイズ | 〜300KB | 500KB | 1MB超 |

運用ルール:
- Firestore read が上限ラインに近づく場合、一覧の再読込回数を削減
- 画像サイズが警戒以上ならクライアント圧縮を必須化

### 5-3. パフォーマンス運用ルール

| 指標 | 目標 | 警戒ライン | 上限ライン |
|---|---:|---:|---:|
| 主要画面表示時間（体感） | 1.0秒以下 | 1.5秒 | 2.0秒超 |
| API/データ取得 p95 | 400ms以下 | 700ms | 1,000ms超 |
| エラー率 | 0.5%以下 | 1.0% | 2.0%超 |

運用ルール:
- 1.5秒超が続く画面は最優先でクエリ改善またはキャッシュ導入
- エラー率 1% 超で新機能より先に安定化対応

### 5-4. 予算超過防止の実行ルール

- [ ] 月初に「今月予算（上限3,000円）」を運用メモに固定
- [ ] 毎週1回、Firebase / Vercel の使用量ダッシュボードを確認
- [ ] 毎週1回、重い画面の read 数をチェック
- [ ] 毎週1回、画像サイズ分布（平均/最大）を確認
- [ ] 月内で 1,500円を超えたら、非必須機能のリリースを凍結して最適化優先
- [ ] 月内で 3,000円到達見込みなら、当月は負荷増機能を停止

## 6) 成長フェーズ別の推奨構成

### Phase 1（現状〜初期公開）
- Vercel Hobby + Firebase（無料中心）
- まずは運用を回して実測データを集める

### Phase 2（利用増加）
- Firebase を従量課金で最小限許容
- 監視・アラートを強化
- read 多発箇所にキャッシュ導入

### Phase 3（中規模）
- 重い処理を非同期化（キュー/バッチ）
- データモデル見直し（ホットスポット回避）
- メディア最適化（圧縮/CDN）
